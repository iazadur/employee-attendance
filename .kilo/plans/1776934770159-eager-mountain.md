# Shift Feature Gap Analysis & Implementation Plan

## Executive Summary

The shift management system is partially implemented but has critical gaps in **frontend CRUD completeness**, **shift assignment workflow**, **input validation**, and **role-based access control**. This plan outlines fixes to deliver a production-ready shift management experience.

---

## Gap Analysis Findings

### Critical Gaps (Blocking Usability)

|---|-----|----------|--------|
| 1 | Frontend shiftsApi missing get/update/delete endpoints | `frontend/src/store/shiftsApi.ts` | Cannot edit or remove shifts from UI |
| 2 | Shifts page lacks edit/delete UI | `frontend/src/app/(app)/shifts/page.tsx` | Admin can only create, not manage existing shifts |

## Proposed Implementation Plan

### Phase 1: Backend Core Fixes (Validation & Data Integrity)

1. **Enhance Shift DTOs with custom validators**
   - Add `@Validate` decorator or custom validation pipes for:
     - Time format `HH:mm` (regex or parsed check)
     - `endTime` must be after `startTime`
     - `workingDays` comma-separated, valid Mon/Tue/Wed/Thu/Fri/Sat/Sun
   - Files: `backend/src/shifts/dto/create-shift.dto.ts`, `update-shift.dto.ts`

2. **Add Prisma-level constraints (safe defaults)**
   - Add `@unique` to `Shift.name`
   - Add check constraints or comments documenting expected formats
   - File: `backend/prisma/schema.prisma`
   - Generate and run migration

3. **Improve Shift service error handling**
   - Distinguish "shift in use" (HTTP 409) vs "not found" (404)
   - Throw appropriate NestJS exceptions
   - File: `backend/src/shifts/shifts.service.ts`

4. **Expand ShiftsController access for MANAGERs (read-only)**
   - Add `@Roles(UserRole.ADMIN, UserRole.MANAGER)` on `GET /shifts` and `GET /shifts/:id`
   - Keep create/update/delete ADMIN-only
   - File: `backend/src/shifts/shifts.controller.ts`

5. **Consider optional shift enforcement in attendance**
   - Option A: Allow check-in with null shift (status PRESENT by default)
   - Option B: Auto-assign a default "General" shift if none
   - Recommendation: Allow check-in, mark as PRESENT, log warning for admin
   - File: `backend/src/attendance/attendance.service.ts`

6. **Expose UpdateEmployee endpoint in frontend API**
   - Add `useUpdateEmployeeMutation` to `employeesApi.ts`
   - File: `frontend/src/store/employeesApi.ts`

7. **Add shift assignment endpoint (optional but clean)**
   - `PATCH /employees/:id/shift` to update only shift
   - Alternative: reuse existing PATCH /employees/:id
   - Recommended: reuse existing update (simpler)

### Phase 2: Frontend CRUD Completion

8. **Inject missing shift API endpoints**
   - Add `useGetShiftQuery`, `useUpdateShiftMutation`, `useDeleteShiftMutation` to `shiftsApi.ts`
   - Add `useUpdateEmployeeMutation` to `employeesApi.ts`

9. **Build shift edit/delete UI on Shifts page**
   - Add Edit button per row → opens pre-filled Dialog
   - Add Delete button per row → confirmation toast → delete
   - Reuse existing form but switch mode (create vs update)
   - File: `frontend/src/app/(app)/shifts/page.tsx`

10. **Improve Employees page shift field UI**
    - Replace text input with `<Select>` dropdown populated from `shifts` query
    - Show loading/error states
    - Allow clearing (no shift)
    - File: `frontend/src/app/(app)/employees/page.tsx`

11. **Add shift selection on employee edit modal**
    - Currently only create dialog; need edit dialog to change shift
    - Use same UpdateEmployee mutation
    - File: `frontend/src/app/(app)/employees/page.tsx` (need edit dialog)

### Phase 3: Reporting & Polish

12. **Add shift filter to Reports page**
    - Dropdown to select shift (or "All")
    - Filter monthly and today KPIs by shift
    - Files: `frontend/src/app/(app)/reports/page.tsx`, `backend/src/reports/reports.service.ts`, `frontend/src/store/reportsApi.ts`

13. **Add shift-based aggregations to Reports service**
    - Extend `todayKpis()` and `monthlyForRequester()` to accept optional `shiftId`
    - Filter attendance records by employee's shift
    - Files: `backend/src/reports/reports.service.ts`

14. **Update reports API slice**
    - Add `shiftId` param to today/monthly queries
    - File: `frontend/src/store/reportsApi.ts`

### Phase 4: Optional Enhancements (if scope allows)

15. **Snapshot shift metadata on attendance check-in**
    - Add `shiftSnapshot` JSON field to `AttendanceRecord` storing shift name/times at check-in
    - Provide audit trail even if shift definition changes
    - File: `backend/prisma/schema.prisma`, `attendance.service.ts`, migration

16. **Add shift deletion cascade/override protection**
    - Prevent deletion if employees assigned
    - Or reassign to "default" shift
    - File: `backend/src/shifts/shifts.service.ts`

17. **Enhance UI with Select components for times/days**
    - Time picker for start/end fields
    - Multi-select for workingDays
    - Shifts page

## Decision Points for User

1. **Shift assignment endpoint**: Use existing employee update (PATCH /employees/:id) or create dedicated `PATCH /employees/:id/shift`?  
   **Recommendation**: Reuse existing - simpler, no extra endpoints.

2. **Null shift handling in attendance**: Allow check-in or require shift?  
   **Recommendation**: Allow check-in, status PRESENT, log warning. Avoid blocking attendance.

3. **Manager shift access**: Read-only or also assign shifts to employees?  
   **Recommendation**: Read-only on shifts; employee edits remain ADMIN-only.

4. **Shift snapshot on attendance**: Store shift metadata at check-in?  
   **Recommendation**: Yes, for historical accuracy if shift details change later.

5. **Scope**: Full implementation (all phases) or minimal viable (critical only)?  
   **Please select**: Full / Critical-only
